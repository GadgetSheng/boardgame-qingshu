# 网页版《情书（Love Letter）》游戏规格说明
1. 文档概述
1.1 文档名称：网页版《情书（Love Letter）》游戏设计规格说明书
1.2 版本：V1.0
1.3 日期：2026-05-17
1.4 目标：本文档旨在定义网页版《情书（Love Letter）》卡牌游戏的核心规则、交互逻辑、数据结构和用户界面要求，为前端开发、后端逻辑和游戏测试提供明确依据。
2. 游戏概述
2.1 游戏简介：网页版《情书》是一款基于经典桌游《Love Letter》的在线多人卡牌推理游戏。玩家扮演公主的追求者，通过出牌、执行卡牌效果来淘汰对手，并在每轮结束时凭借手牌点数（亲密度）获胜，最终积累足够“爱意”赢得游戏。
2.2 游戏模式：

多人匹配模式：支持2-6名玩家在线实时对战。
创建房间模式：玩家可创建私人房间，邀请好友加入，自定义胜利条件。
AI练习模式：玩家可与1-5个AI机器人进行对战。

3. 核心游戏规则（逻辑层）​
3.1 游戏组件与初始化

牌库：使用标准16张卡牌组（可扩展支持32张）。卡牌数据模型需包含：cardId（唯一标识）、name（名称）、value（点数1-8）、count（数量）、effectDescription（效果文本）、effectLogic（效果执行函数标识）。
初始化流程：

服务器创建游戏房间，根据玩家人数确定胜利所需爱意指示物数量（见3.5）。
洗牌算法生成随机牌库序列。
暗置移除：从牌库顶部移除一张牌，放置于“移除区”，本局游戏不再使用。此牌信息对所有人不可见。
为每位玩家从牌库顶部抽取一张牌作为初始手牌。
随机决定起始玩家。



3.2 游戏回合流程（状态机）​
游戏应维护一个明确的回合状态机，包含以下状态：WAITING_FOR_TURN -> DRAW_PHASE -> ACTION_PHASE -> EFFECT_RESOLUTION -> TURN_END。

抽牌阶段 (DRAW_PHASE)：当前回合玩家从牌库顶部自动抽取一张牌加入手牌。如果牌库已空，则触发“回合结束阶段”。
行动阶段 (ACTION_PHASE)：玩家从当前两张手牌中选择一张打出。界面需高亮可选择的牌。
效果结算阶段 (EFFECT_RESOLUTION)：

客户端向服务器发送出牌指令（cardId, targetPlayerId（如有需要）, guessCardId（如守卫猜测））。
服务器验证操作合法性后，广播执行卡牌效果。
效果必须强制执行，服务器需处理所有效果逻辑（如比较手牌、弃牌、查看手牌、玩家淘汰等）。
执行效果可能改变游戏状态（如玩家出局），需立即更新所有客户端。


回合结束阶段 (TURN_END)：

打出的牌移至该玩家的“公开弃牌堆”。
回合权按顺时针顺序移交给下一位未被淘汰的玩家。
如果牌库在抽牌阶段被抽空，则当前玩家完成其回合后，游戏进入轮次结算阶段。



3.3 玩家淘汰与状态

淘汰条件：由卡牌效果触发（如守卫猜中、男爵比输、主动打出公主）。
淘汰后行为：客户端界面标记该玩家为“已出局”。其手牌自动公开（但不触发效果），后续回合被跳过。
数据同步：服务器需维护一个isEliminated的玩家状态，并同步给所有客户端。

3.4 一轮游戏结束与结算

触发条件：
a) 牌库抽空，且当前回合结束。
b) 场上仅剩一名玩家未被淘汰。
结算逻辑：

所有未被淘汰的玩家自动亮出手牌。
服务器比较所有幸存玩家的手牌value（点数）。
获胜判定：点数最高的玩家赢得本轮。平局处理：若最高点数出现并列，则所有并列玩家同时获胜（各获得一个爱意指示物）。
获胜玩家（们）的affectionToken（爱意指示物）计数+1。
广播本轮结果（获胜者、各自手牌、获得的爱意指示物）。
进入下一轮初始化（重新洗牌、发牌，由本轮获胜者或随机决定起始玩家）。



3.5 游戏最终胜利条件

服务器需根据游戏开始时的玩家人数，设定最终胜利所需爱意指示物数量。建议采用经典规则：





























玩家人数所需爱意指示物2人7个3人5个4人4个5人4个6人3个

当任何玩家的affectionToken达到或超过目标值时，游戏立即结束，该玩家获得最终胜利。界面应显示胜利画面和统计。

4. 卡牌效果详细规格（需后端实现）​
每张卡牌的效果必须由服务器权威计算，以下是关键效果的技术描述：


守卫 (1) - 猜测：

输入：targetPlayerId, guessCardValue（猜测的点数，不能是1）。
逻辑：检查目标玩家当前手牌的value是否等于guessCardValue。
结果：若相等，则淘汰目标玩家；否则无效果。



男爵 (3) - 秘密比较：

输入：targetPlayerId。
逻辑：服务器比较当前玩家与目标玩家的手牌value。
结果：点数较低的玩家被淘汰。平局：双方均不被淘汰。



王子 (5) - 强制弃牌重抽：

输入：targetPlayerId（可以是自己）。
逻辑：目标玩家弃置其当前手牌（若弃置的是公主(8)，则触发淘汰）。然后，若牌库还有牌，目标玩家立即从牌库顶抽取一张新牌；若牌库已空，则无法抽牌，目标玩家进入无手牌状态（将在结算时直接判负）。



国王 (6) - 交换手牌：

输入：targetPlayerId。
逻辑：交换当前玩家与目标玩家的手牌对象。



伯爵夫人 (7) - 强制弃牌规则：

触发时机：在玩家的行动阶段(ACTION_PHASE)选择手牌时。
客户端规则：若玩家手牌中包含伯爵夫人(7)，且另一张手牌是国王(6)或公主(8)，则必须高亮伯爵夫人作为唯一可选项，或禁止选择另一张牌。
服务器验证：收到出牌请求时，需验证此规则。若玩家违规打出国王或公主，服务器应拒绝并提示。



5. 用户界面与交互要求（前端层）​
5.1 游戏主界面布局

中央区域：牌库（显示剩余牌数）、移除区（牌背朝上）。
玩家区域：为每个玩家（包括自己）设置固定区域，显示：玩家头像/名称、爱意指示物数量、状态（进行中/已出局）、公开弃牌堆（牌面朝上叠放）。
手牌区（自己）：清晰展示自己的手牌（牌面信息仅自己可见）。在行动阶段，可选择的牌应有悬停或高亮效果。
信息流/聊天区：显示系统消息（如“A玩家打出了守卫，猜测B玩家手牌是王子”）、回合提示、效果结算结果。

5.2 关键交互流程

出牌操作：点击手牌 -> 若卡牌需要目标（如守卫、男爵、王子、国王），弹出玩家选择列表 -> 若为守卫，需额外弹出猜测选择器（选择2-8的点数）-> 确认提交。
查看信息：鼠标悬停在任意玩家的公开弃牌堆上，可展开查看其历史打出牌的顺序。
回合与状态提示：清晰标注当前回合玩家，并给出文字提示（如“请抽牌”、“请选择一张牌打出”）。

5.3 动画与反馈

抽牌/出牌动画：牌从牌库移动到手牌区，或从手牌区移动到弃牌堆，应有平滑的移动动画。
效果结算反馈：关键效果（如玩家被淘汰、交换手牌、查看手牌<仅对执行者显示>）应有明显的视觉和文字反馈。
声音提示：可配置不同的音效用于抽牌、出牌、淘汰、获胜等事件。

6. 数据通信协议（示例）​
6.1 客户端 -> 服务器

PLAY_CARD: { action: "play_card", cardId: "guard_1", targetPlayerId: "player2", guess: 5 }
READY_FOR_NEXT_ROUND: { action: "ready" }

6.2 服务器 -> 客户端（广播）​

GAME_STATE_UPDATE: { phase: "ACTION_PHASE", currentPlayer: "player1", deckCount: 5, players: [...] }
CARD_EFFECT_RESULT: { card: "guard", playedBy: "player1", target: "player2", guess: 5, success: true, eliminated: "player2" }
ROUND_END: { winners: ["player1"], revealedHands: { player1: { card: "king", value: 6 }, ... }, tokensAwarded: { player1: 1 } }
GAME_OVER: { winner: "player3", finalTokens: 7 }
